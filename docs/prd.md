# Earnest Ice Cream Flavor Explorer - Product Requirements Document

## Overview

Earnest Ice Cream Flavor Explorer is an unofficial, fan-made static website for exploring current Earnest Ice Cream flavors across four Vancouver-area scoop shops. It helps customers answer a practical question quickly: which flavors are available as scoops, pints, or pre-made sandwiches at each location?

The app is a single-page React experience deployed as static files. Flavor data is scraped from Earnest's public location pages and stored in the repository as generated JSON.

## Goals

- Show current flavor availability across Fraser Street, Quebec Street, Frances Street, and North Vancouver.
- Preserve the difference between scoop, pint, and sandwich availability.
- Make it fast to filter by serving type, location, search query, and vegan status.
- Keep generated data stable through scraper validation and automated tests.
- Deploy without a backend.

## Non-Goals

- This site is not affiliated with Earnest Ice Cream.
- It does not guarantee real-time in-store inventory.
- It does not provide ordering, reservations, accounts, favorites, or payment.
- It does not scrape ingredient or allergen details beyond the vegan indicator inferred from flavor names.

## Users

Primary users are customers deciding which Earnest location to visit or whether a flavor is available as a scoop, pint, or sandwich. Secondary users are maintainers who need to update scraper logic when Earnest changes page markup.

## Functional Requirements

### Flavor Browsing

- The app must render all flavors that match the active filters.
- Flavor cards must sort alphabetically by display name.
- Each card must show the flavor name.
- Vegan flavors must show a vegan indicator.
- When a serving type is selected, card location badges must reflect availability for that serving type only.
- Clicking a flavor card details action must open the flavor detail panel.

### Search

- Search must be case-insensitive.
- Search must be punctuation-insensitive for common flavor text such as `&`, `+`, and repeated whitespace.
- Multiple search tokens must all match the normalized flavor name.
- Empty search must not filter results.

### Serving Type Filter

- Default serving type must be `scoop`.
- Serving type options must be visible without opening a dropdown.
- Exactly one serving type must be active at a time.
- Options: `scoop`, `pint`, `sandwich`.
- Filtered results must include only flavors available for the selected serving type.

### Location Filter

- Default location must be `all`.
- Location options must be visible without opening a dropdown.
- Exactly one location option must be active at a time.
- `all` must include all stores.
- A specific location must include only flavors available at that location for the selected serving type.

### Vegan Filter

- Vegan-only must be off by default.
- When enabled, results must include only flavors with `isVegan: true`.
- The vegan control should read visually green in light and dark mode.

### Clear Filters

- The clear action must reset search to empty, location to `all`, serving type to `scoop`, and vegan-only to false.
- The clear action should appear only when non-default filters are active.

### Theme

- Light and dark themes must be supported.
- Theme selection must persist in localStorage.
- Light mode must use sufficient contrast for text, controls, borders, and location badges.

## Data Requirements

### Flavor Type

```typescript
interface Flavor {
  id: string;
  name: string;
  isVegan: boolean;
  scoopStores: StoreId[];
  pintStores: StoreId[];
  sandwichStores: StoreId[];
  description: string;
}
```

Important data rules:

- `scoopStores`, `pintStores`, and `sandwichStores` are independent.
- A flavor may be available as a pint at a store where it is not available as a scoop.
- Sandwich-only items have empty `scoopStores` and `pintStores`.
- `id` is a stable slug generated from the flavor name.
- `isVegan` is inferred from flavor names containing `vegan`.
- Descriptions are supplied by scraper-side known descriptions or fallback text.

### Store Type

```typescript
type StoreId = 'fraser' | 'quebec' | 'frances' | 'northvan';

interface Store {
  id: StoreId;
  name: string;
  shortName: string;
  address: string;
  phone: string;
  hours: string;
  description: string;
  color: string;
  bgColor: string;
}
```

## Scraper Requirements

The scraper lives at `scripts/scrape-flavors.js`.

It must:

- Visit the four configured Earnest location URLs.
- Scrape `Scoops`, `Pints`, and `Sandwiches` separately.
- Prefer the clicked tab's target panel to avoid mixing hidden tab content.
- Fall back to active or visible flavor links when target panel lookup is unavailable.
- Normalize, filter, and deduplicate flavor link text.
- Build unified flavor objects with independent scoop, pint, and sandwich store arrays.
- Refuse to overwrite generated data when a store returns no scoops or no pints.
- Reject duplicate normalized names within a serving category for a store.
- Write `src/data/flavors.json` only after validation passes.

## Testing Requirements

The test suite uses TypeScript compilation plus Node's built-in test runner.

Current test coverage:

- Generated flavor data has required fields and valid store references.
- Data helpers return serving-appropriate results.
- Current checked-in sandwich inventory is pinned.
- Search and filter logic works with sample data.
- Store serving availability handles split scoop/pint availability.
- Scraper parsing handles link filtering, hidden tab content, visible-link fallback, serving separation, and validation failures.

`npm run build` must run tests before TypeScript and Vite production builds.

## Automation and Deployment

The GitHub Actions workflow `.github/workflows/update-flavors.yml` runs daily at 6:00 AM UTC and can also be triggered manually.

Workflow behavior:

1. Checkout repository.
2. Setup Node.js 20.
3. Install dependencies with `npm ci`.
4. Install Playwright Chromium.
5. Run `node scripts/scrape-flavors.js`.
6. Commit `src/data/flavors.json` if it changed.
7. Run `npm run build`.
8. Deploy `dist/` to GitHub Pages.

## Acceptance Criteria

- A user can find a flavor by search.
- A user can switch between scoop, pint, and sandwich availability without a dropdown.
- A user can filter to a specific store without a dropdown.
- A user can enable vegan-only filtering.
- A user can open flavor details and see serving availability by store.
- The app builds as static assets.
- The scraper can update `src/data/flavors.json` without manual editing.
- Automated tests pass after scraper updates.
